Day 1 · Cycle 1 · Eval Frameworks
Format: BUILD

Building on: First time on this topic.

Concept (2–3 sentences)
Eval frameworks give you a repeatable, quantitative signal on whether your LLM system is getting better or worse — without waiting for users to complain. Offline evals run against a fixed golden dataset; LLM-as-judge replaces brittle regex checks with a model that scores outputs on dimensions humans actually care about (helpfulness, accuracy, tone). In production, this loop is what separates teams that ship confidently from teams that guess.

The 60-minute task

Build a minimal offline eval harness in Python for a support-ticket triage classifier.

What you're building: A script that classifies incoming support tickets into categories (billing, technical, account, other), then evaluates those classifications with an LLM-as-judge.

Stack: Python 3.11+, anthropic SDK, no external eval library (build it from scratch so you understand the primitives).

Step-by-step scope:

Golden dataset (10 min) — Hard-code 15 support ticket examples as a list of dicts:
GOLDEN = [
  {"ticket": "My invoice is wrong, charged twice", "expected": "billing"},
  {"ticket": "App crashes on login on iOS 17",    "expected": "technical"},
  # ... 13 more covering all 4 categories, ~3–4 per class
]

Classifier (10 min) — Call claude-haiku-4-5 with a short system prompt. Return the predicted category as a JSON field so you can parse it reliably.

Exact-match scorer (5 min) — Compare predicted vs expected. Compute accuracy (%).
LLM-as-judge (20 min) — For each ticket where the classifier got it right, ask claude-haiku-4-5 a second time to rate confidence/quality on a 1–5 scale with a brief rationale. For misclassifications, ask the judge why it thinks the classifier was wrong. Collect scores.

Report (10 min) — Print a structured summary:
Accuracy:        13/15 (86.7%)
Judge avg score: 4.1 / 5.0
Failures:
  - "Can't reset 2FA" → predicted: account, expected: technical
    Judge: "2FA is an auth/technical issue, not account management"

Wrap-up (5 min) — Write one findings.md paragraph: what patterns caused failures, and what prompt change would fix them.

Definition of done:
Script runs end-to-end without errors
Produces accuracy metric + per-failure judge rationale
findings.md contains at least one actionable prompt-improvement hypothesis

Stretch (if time allows or weekend revisit)
Add a second "model variant" — run the same golden set through claude-sonnet-4-6 and compare accuracy + judge scores. Build a simple compare_runs() function that prints a side-by-side diff. This is the seed of a real eval regression suite.

Interview hook (60-second answer)
"I built a from-scratch offline eval harness for a support-ticket classifier — no frameworks, just the raw primitives. It ran 15 golden examples through Claude Haiku, scored exact-match accuracy, then used a second LLM-as-judge call to rate quality and explain failures. The insight was that LLM-as-judge catches why a failure happened — not just that it happened — which turns a pass/fail number into an actionable prompt fix. That's the difference between an eval that tells you your score and one that tells you what to do next."
