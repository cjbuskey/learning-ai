The Eval Harness Playbook

A Mets Fan's Guide to LLM Evaluation

First Pitch: Why Evals?

Imagine you're the Mets' manager. You want to know if your starting pitcher is getting better or worse — you don't wait for fans to boo from the upper deck. You look at stats: ERA, WHIP, strikeout rate.

An eval framework is the same idea for LLMs. Instead of waiting for users to complain, you run your model against a fixed set of test cases and measure how it does. Change the prompt? Run the eval. New model version? Run the eval. Did the score go up or down? That's your scoreboard.

The core idea: evals turn "I think the model is good" into a number you can defend.

The Lineup: How Our Harness Bats

This script bats five hitters in order. Each one does one job.

1

Leadoff — The Golden Set

A hand-written list of 15 support tickets, each with the correct category already known (billing, technical, account, other). This is your "answer key." Without it, you can't score anything.
2

Two-Hole — The Classifier

Sends each ticket to claude-haiku-4-5 with a system prompt that says "pick one of these four categories, return JSON." That's the system under test — the player whose stats we care about.
3

Three-Hole — The Exact-Match Scorer

Did the predicted category equal the expected category? Yes or no. Add them up, divide by total → accuracy. Simple, but this is the official box score.
4

Cleanup — The LLM-as-Judge

Here's the trick. We send each result to Claude a second time and ask: "Was this an obvious correct answer (5/5) or a lucky guess (1/5)? If wrong, why?" Regex can't do this — it has no taste. An LLM can.
5

Five-Hole — The Report

Print the scoreboard, dump results.json, and surface failures with the judge's reasoning. That last bit — the why behind a failure — is the actionable part.

The Actual Code: What Runs When You Type python eval.py

There is exactly one script in this project: eval.py. No frameworks, no test runner — just Python calling the Anthropic SDK in a loop. Here's a tour, in execution order.

① Bootstrap

from anthropic import Anthropic
from dotenv import find_dotenv, load_dotenv

load_dotenv(find_dotenv())   # walks up to repo root, loads .env
CLIENT = Anthropic()          # auto-reads ANTHROPIC_API_KEY
MODEL = "claude-haiku-4-5"
CATEGORIES = ["billing", "technical", "account", "other"]

The find_dotenv() call is what lets a single shared .env at the repo root work for every project folder.

② The Golden Set (the answer key)

GOLDEN = [
    {"ticket": "My invoice is wrong, charged twice...", "expected": "billing"},
    {"ticket": "App crashes on login on iOS 17...",     "expected": "technical"},
    # ...13 more
]

③ _call_json(system, user) — the shared API helper

def _call_json(system, user):
    msg = CLIENT.messages.create(
        model=MODEL, max_tokens=300,
        system=system,
        messages=[{"role": "user", "content": user}],
    )
    text = msg.content[0].text.strip()
    # strip ``` fences if the model added them
    return json.loads(text)

Both the classifier and the judge go through this function — the only difference between them is the system prompt.

④ classify(ticket) — API call #1

def classify(ticket):
    out = _call_json(CLASSIFIER_SYSTEM, f"Ticket: {ticket}")
    return out.get("category", "other")

Returns one of the 4 category strings. This is the system under test.

⑤ judge(result) — API call #2

def judge(result):
    user = f"Ticket: {result.ticket}\nExpected: {result.expected}\nPredicted: {result.predicted}"
    if result.correct:
        out = _call_json(JUDGE_CORRECT_SYSTEM, user)   # rate 1-5
        result.judge_score = int(out["score"])
    else:
        out = _call_json(JUDGE_WRONG_SYSTEM, user)     # explain miss
    result.judge_rationale = out.get("rationale", "")

This is the LLM-as-judge — Claude grading Claude. Different prompt depending on whether the classifier got it right.

⑥ run() — the main loop

def run():
    results = []
    for i, ex in enumerate(GOLDEN, 1):
        predicted = classify(ex["ticket"])             # API call #1
        r = Result(ticket=..., expected=..., predicted=...,
                   correct=(predicted == ex["expected"]))   # scorer (inline)
        judge(r)                                        # API call #2
        results.append(r)
    return results

Each iteration = 2 API calls. 15 tickets × 2 = 30 calls per run. Cheap on Haiku (well under the $5 cap).

⑦ report(results) — the box score

def report(results):
    correct = [r for r in results if r.correct]
    acc = len(correct) / len(results) * 100
    avg = sum(r.judge_score for r in correct) / len(correct)
    print(f"Accuracy:        {len(correct)}/{len(results)} ({acc:.1f}%)")

⑧ The entry point

if __name__ == "__main__":

The At-Bat: One Ticket, Step by Step

Here's what happens to a single ticket like "App crashes on login on iOS 17":

1. CLASSIFIER CALL
   system: "Classify into billing/technical/account/other..."
   user:   "Ticket: App crashes on login on iOS 17"
   →  {"category": "technical"}

2. SCORER
   expected = "technical"
   predicted = "technical"
   correct = True   ✓

3. JUDGE CALL  (because it was correct)
   system: "Rate clarity 1-5..."
   →  {"score": 5, "rationale": "Crash + iOS version is unambiguously technical"}

4. RESULT STORED
   Result(ticket=..., correct=True, judge_score=5, ...)

Multiply that by 15 tickets, and you have a full game's worth of at-bats. Each one produces a row in your results — a stat line you can compare across runs.

Now we have signal. The first run hit 100% — a warning sign that the eval was too easy. After adding 4 adversarial tickets (keyword traps, mixed intent, ambiguous overlap), the model misses 3 of them. That's the eval doing its job: a 100% score told us nothing; an 84.2% score with reasoned failures tells us exactly where the prompt is weak.

The 3 failures

TicketPredictedExpectedJudge's reasoning
"I was charged again after I deleted my account last month — please close it for good." billingaccount The charge is mentioned, but the ask is account closure.
"Is there a way to bulk-export tickets older than 90 days? Couldn't find it in the docs." technicalother Feature/docs inquiry, not a bug report.
"Do your invoices show line-item taxes? Our finance team needs that for compliance." billingother Product capability question, not a billing dispute.

Pattern: the classifier latches onto keywords ("charged", "export", "invoice") instead of the user's primary ask. That's a one-line prompt fix waiting to happen — and now we have a way to measure whether it works.

Why "LLM-as-Judge" Beats Regex

ApproachWhat it seesBaseball analogy
Exact match / regex Did the string match? Pass/fail only. Box score: hit or out. No nuance.
LLM-as-judge Quality of the answer + why it was right or wrong. Color commentary: "He chased a slider low and away — pitcher set him up perfectly."

The judge turns a pass/fail number into a diagnosis. That's the difference between a stat sheet and a coach who can tell you what to fix.

Next at the Plate (Cycle 2)

The harness works. The signal doesn't. Next cycle on this topic:

  • Throw harder pitches. Add adversarial tickets — ambiguous overlaps, keyword traps, sparse signal.
  • Compare two pitchers. Run the same golden set through Sonnet 4.6 and diff the scores.
  • Track the trend. Save runs over time so prompt changes can be regression-tested.
The interview hook: "I built an eval, scored 100%, and the real lesson was the eval — not the model. A golden set without near-misses can't discriminate between prompts, so the next iteration deliberately seeds boundary cases."
